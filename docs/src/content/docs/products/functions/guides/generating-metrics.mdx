The [Metrics](/products/functions/metrics) tab only has something to show once a function has received traffic — and the more your requests vary, the more of the tab you can see. A single `GET` that always returns `200` leaves the status, error-rate, and response-time charts flat.

This guide deploys a **sample function** that deliberately varies its HTTP method, status code, latency, and response size, then drives some traffic to it so every chart on the Metrics tab lights up.

:::caution
This is a demonstration function. Remove it once you have real traffic — it exposes an endpoint that can return errors and add artificial latency on demand.
:::

## Create the Function

<Tabs>
  <TabItem label="TypeScript">

```ts title="./functions/sample.ts"
import type { Request, Response } from 'express'

export default async (req: Request, res: Response) => {
  // Optional artificial latency: /sample?delay=250 (milliseconds, capped at 2s)
  const delay = Math.min(Number(req.query.delay) || 0, 2000)
  if (delay > 0) {
    await new Promise((resolve) => setTimeout(resolve, delay))
  }

  // Optional forced error: /sample?fail=true -> 500
  if (req.query.fail === 'true') {
    res.status(500).json({ error: 'Simulated failure' })
    return
  }

  switch (req.method) {
    case 'POST':
      res.status(201).json({ created: true, received: req.body })
      return
    case 'DELETE':
      res.status(204).end()
      return
    case 'GET':
      res.status(200).json({ items: Array.from({ length: 10 }, (_, i) => i) })
      return
    default:
      res.status(405).json({ error: `Method ${req.method} not allowed` })
  }
}
```

Deploy it by pushing to your connected GitHub repository, as described in [Getting Started](/products/functions/guides/getting-started).

## Generate Traffic

Send a spread of requests so the charts have varied data. Replace the URL with your function's endpoint:

```shell
URL="https://[subdomain].functions.[region].nhost.run/v1/sample"

for i in $(seq 1 50); do
  curl -s "$URL" > /dev/null                                       # 200 GET
  curl -s -X POST "$URL" -H 'Content-Type: application/json' \
    -d "{\"n\": $i}" > /dev/null                                   # 201 POST
  curl -s -X DELETE "$URL" > /dev/null                            # 204 DELETE
  curl -s "$URL?delay=200" > /dev/null                            # slower 200
  if [ $((i % 5)) -eq 0 ]; then
    curl -s "$URL?fail=true" > /dev/null                          # 500 error
  fi
done
```

## View the Results

Open the [Metrics](/products/functions/metrics) tab for the `sample` function. With the traffic above you should now see:

- **Invocations** split across `GET`, `POST`, and `DELETE`.
- **Response Status** showing a mix of `200`, `201`, `204`, and `500`.
- **Error Rate** rising above zero from the forced failures.
- **Response Times** reflecting the slower `?delay=200` calls.
- **Average Response Size** differing per method (`DELETE` returns no body; `POST` echoes the request).
