Agents are conversational AI interfaces that can interact with users, use tools, and stream responses in real time. Unlike [Assistants](/products/ai/assistants), which are built on top of the OpenAI Assistants API, Agents are a provider-agnostic solution that supports multiple LLM providers and offers real-time streaming via Server-Sent Events (SSE).

## Key Features

- **Multi-provider support** â€” Use Anthropic (Claude), OpenAI, or Google Gemini as the underlying LLM
- **Real-time streaming** â€” Responses are streamed token-by-token via SSE
- **Tool use** â€” Agents can use tools like web search, web fetch, and MCP (Model Context Protocol) servers, with optional per-tool human approval
- **Session management** â€” Conversations are organized into sessions with full message history persistence
- **Agentic loop** â€” Agents can autonomously chain multiple tool calls before responding (up to 25 iterations per message)

## Providers

Agents support three LLM providers. You only need to configure keys for the providers you intend to use.

| Provider | Environment Variable | Example Models |
|---|---|---|
| Anthropic (Claude) | `ANTHROPIC_API_KEY` | `claude-sonnet-4-20250514`, `claude-haiku-4-20250414` |
| OpenAI | `OPENAI_API_KEY` | `gpt-4o`, `gpt-4o-mini` |
| Google Gemini | `GOOGLE_AI_API_KEY` | `gemini-2.0-flash`, `gemini-2.5-pro-preview-06-05` |

You can pass any model identifier supported by the provider's API.

## Configuration

To use agents, configure the API key for your chosen provider(s) in your project's configuration:

```toml
[ai]
version = '0.6.0'
webhookSecret = '{{ secrets.GRAPHITE_WEBHOOK_SECRET }}'

[ai.openai]
apiKey = '{{ secrets.OPENAI_API_KEY }}'
```

Additional provider keys are configured via environment variables (`ANTHROPIC_API_KEY`, `GOOGLE_AI_API_KEY`). If an agent is configured with a provider whose API key is not set, requests will return an error.

## Permissions

Agent management (CRUD) and session/message operations are handled directly by Hasura's auto-generated GraphQL API. This means Hasura's permission rules on the underlying tables control what each user can access. The following tables need permissions configured for the `user` role:

#### `graphite.agents`

| Operation | Columns | Notes |
|---|---|---|
| select | `id`, `name`, `description`, `instructions`, `provider`, `model`, `tools_config`, `created_at`, `updated_at` | Required for listing agents and retrieving agent details |
| insert | `user_id`, `name`, `description`, `instructions`, `provider`, `model`, `tools_config` | Required for creating agents |
| update | `name`, `description`, `instructions`, `provider`, `model`, `tools_config`, `updated_at` | Required for updating agents |
| delete | â€” | Required for deleting agents |

#### `graphite.agent_sessions`

| Operation | Columns | Notes |
|---|---|---|
| select | `id`, `agent_id`, `user_id`, `created_at`, `updated_at` | Required for listing sessions and retrieving session details |
| insert | `agent_id`, `user_id` | Required for starting a new session |
| delete | â€” | Required for deleting a session |

#### `graphite.agent_messages`

| Operation | Columns | Notes |
|---|---|---|
| select | `id`, `session_id`, `role`, `content`, `tool_calls`, `tool_call_id`, `tool_name`, `created_at` | Required for retrieving session message history |

:::note
The SSE streaming endpoint (`POST /v1/agents/sessions/{sessionID}/messages`) and the tool approval endpoint (`POST /v1/agents/sessions/{sessionID}/approve-tools`) both authenticate every request. Users must pass an `Authorization: Bearer <token>` header; the service queries Hasura with the user's credentials to verify session access based on the permission rules configured on `graphite.agent_sessions`. Successful checks are cached so subsequent messages in the same session avoid repeated Hasura queries. Admin access is also supported via the `X-Hasura-Admin-Secret` header. Message persistence uses an internal service connection, so no Hasura permissions on `agent_messages` insert are required.
:::

### Restricting Access

You can use Hasura's permission rules to restrict access further. Some common patterns:

- **Users can only manage their own agents** â€” Add a filter on `agents` select/update/delete: `{ "user_id": { "_eq": "X-Hasura-User-Id" } }` and set `user_id` as a column preset on insert
- **Users can only see their own sessions** â€” Add a filter on `agent_sessions` select: `{ "user_id": { "_eq": "X-Hasura-User-Id" } }`
- **Users can only delete their own sessions** â€” Add a filter on `agent_sessions` delete: `{ "user_id": { "_eq": "X-Hasura-User-Id" } }`
- **Restrict which agents are visible** â€” Add a filter on `agents` select to limit agents by name, provider, or any other column
- **Users can only see messages from their sessions** â€” Add a filter on `agent_messages` select joining through `agent_sessions.user_id`

## Agents vs Assistants

| Feature | Agents | [Assistants](/products/ai/assistants) |
|---|---|---|
| LLM Providers | Anthropic, OpenAI, Google | OpenAI only |
| Streaming | Real-time SSE | No (synchronous responses) |
| Tools | Web search, web fetch, GraphQL, MCP (with optional approval) | GraphQL queries/mutations, webhooks, file stores |
| API | REST (SSE) + Hasura GraphQL (management) | GraphQL only |
| Data access | Via configured tools | Direct access to project data via GraphQL |
| Session storage | Built-in (PostgreSQL) | OpenAI-managed threads |
