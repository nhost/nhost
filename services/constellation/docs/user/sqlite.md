# SQLite source differences

Constellation exposes SQLite databases through the same Hasura-compatible GraphQL
metadata model used for PostgreSQL sources where SQLite can support the same
semantics. Some PostgreSQL features have no SQLite equivalent; this page records
the customer-visible differences that follow from those limits.

## Non-admin upserts with permission checks

For a non-admin `insert` / `insert_one` mutation with `on_conflict` and non-empty
`update_columns`, Constellation enforces both sides of the permission model:

- rows that are inserted must satisfy the role's INSERT `check`; and
- rows that take the `DO UPDATE` branch must satisfy the role's UPDATE `filter`
  and UPDATE `check`.

PostgreSQL can classify each returned upsert row exactly by using the internal
`xmax` value in `RETURNING`, matching Hasura's behavior even when the conflict
key is generated by the database or supplied by a default.

SQLite does not expose an equivalent per-row upsert action marker. For SQLite
sources Constellation can still classify rows when every conflict-target column
is present in the insert input: it pre-selects the rows that already exist and
uses that key set to separate inserts from updates.

If the conflict key cannot be known from the input — for example, a conflict
column is omitted and populated by a SQLite default or generated value —
Constellation fails closed. The UPDATE check is evaluated against every row
returned by the upsert, and any post-mutation INSERT check that could not be run
before the INSERT is also evaluated against every returned row. This can reject a
pure insert that PostgreSQL/Hasura would accept, but it avoids allowing an
undetectable SQLite `DO UPDATE` row to bypass the role's UPDATE check.

When this fallback rejects the mutation, the error is the normal insert/update
permission failure: `check constraint of an insert/update permission has failed`.
