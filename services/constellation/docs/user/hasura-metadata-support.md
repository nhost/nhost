# Hasura Metadata Support

Constellation consumes Hasura-compatible metadata but implements only the parts
that run on the GraphQL request path. This document maps **every Hasura metadata
field** to its status in Constellation so you can tell, before switching over,
exactly what carries across and what is dropped.

It is generated by reading Constellation's metadata parser (`metadata/`) and the
code that actually consumes each field (`connector/sql/graphql/schema`,
`connector/sql/graphql/queries`, `connector/remoteschema`, `connector/customization`,
`connector/`), and comparing the result against Hasura's
[Metadata API reference](https://hasura.io/docs/latest/api-reference/metadata-api/index/).

## Legend

| Status | Meaning |
|---|---|
| тЬЕ **Supported** | Parsed **and** consumed. Behaves like Hasura (modulo the intentional divergences in [`KNOWN_DIFFERENCES.md`](../../KNOWN_DIFFERENCES.md)). |
| ЁЯЯб **Partial** | Works, but only in some forms or with caveats noted inline. |
| тЪк **Ignored** | The surrounding object is parsed, but this field is **silently dropped** тАФ no effect, no error. It looks configured but does nothing. |
| тЭМ **Unsupported** | The whole feature area is not modeled. The metadata key is never read. |

> **The important caveat about тЪк and тЭМ:** Constellation does **not** reject
> unknown metadata. There is no strict/`disallow_unknown_fields` mode. A real
> Hasura `metadata.json` containing actions, event triggers, cron triggers,
> allowlists, etc. will **load without error** тАФ those features simply will not
> exist in the served API. The same is true field-by-field: a `limit` on a select
> permission, or a `pool_settings` block, is accepted and thrown away. Treat
> тЪк/тЭМ rows as *silently inert*, not *rejected*. (One partial exception: on the
> DB metadata source, event-trigger config is *stored* and round-trips through
> `export_metadata` even though no delivery runtime fires the triggers тАФ see
> [Event triggers](#entirely-unsupported-feature-areas).)

## How Constellation reads Hasura metadata

Three input shapes, one effective support surface (they all funnel through the
same converter in `metadata/convert.go`):

| Mode | Source | Notes |
|---|---|---|
| **File (Hasura v3 directory)** | `--metadata-path` pointing at a Hasura metadata dir | Reads **only** `<root>/databases/databases.yaml` and the optional `<root>/remote_schemas.yaml`. `!include` directives are followed. **No other file is opened** тАФ `actions.yaml`, `cron_triggers.yaml`, `query_collections.yaml`, `allow_list.yaml`, `rest_endpoints.yaml`, `inherited_roles.yaml`, etc. are never read. |
| **Database (polled)** | `--metadata-database-url` тЖТ `hdb_catalog.hdb_metadata` | Parses the JSON blob Hasura stores. Must be `version: 3`. The blob keys its source list as `sources` (handled). Unknown top-level keys are dropped. |
| **Native TOML** | `--metadata-path` ending in `.toml` | Constellation's own format. Same shape as the tables below; no Hasura-only keys exist to drop. |

## Sources / database connection

```yaml
- name: default
  kind: postgres
  configuration:
    connection_info:
      database_url:
        from_env: HASURA_GRAPHQL_DATABASE_URL   # or an inline "postgres://..." string
  tables: "!include default/tables/tables.yaml"
  functions: "!include default/functions/functions.yaml"
```

| Field | Status | Notes |
|---|---|---|
| `name` | тЬЕ | Source name; used for cross-source remote relationships. |
| `kind` | ЁЯЯб | Only `postgres` and `sqlite` are accepted. Any other value (`citus`, `mssql`, `bigquery`, `pg`, тАж) fails at startup with `unsupported database kind: <kind>`. `sqlite` is a Constellation extension, not a Hasura backend. |
| `configuration.connection_info.database_url` | тЬЕ | Literal string or `{ from_env: VAR }`. |
| `tables` | тЬЕ | Inline list or `!include`. |
| `functions` | тЬЕ | Inline list or `!include`. Postgres only (gated by `SupportsFunctions`). |
| `configuration.connection_info.pool_settings` | тЪк | `max_connections`, `idle_timeout`, `retries`, `pool_timeout`, `connection_lifetime` are **not read**. Tune the pool via connection-string params instead (pgx parses `?pool_max_conns=тАж&pool_max_conn_lifetime=1h&pool_max_conn_idle_time=тАж` etc.). Constellation also applies its own minimum floors. |
| `configuration.connection_info.use_prepared_statements` | тЪк | Dropped. |
| `configuration.connection_info.isolation_level` | тЪк | Dropped. |
| `configuration.connection_template` | тЪк | Dynamic connection routing is dropped. |
| `configuration.connection_set` | тЪк | Dropped. |
| `configuration.read_replicas` | тЪк | No read-replica routing. |
| `configuration.extensions_schema` | тЪк | Dropped. |
| `customization.root_fields` (`namespace`, `prefix`, `suffix`) | тЬЕ | Source-level GraphQL customization. `namespace` wraps every root field under a single field (named `<namespace>`) on each operation type; `prefix`/`suffix` are applied to root field names. |
| `customization.type_names` (`prefix`, `suffix`) | тЬЕ | Prepended/appended to every non-builtin type name. Scalars, the `order_by` enum, and `*_comparison_exp` inputs are deliberately left uncustomized to match Hasura, so they still dedup across sources. (`mapping` is remote-schema-only; ignored for databases.) |
| `customization.naming_convention` | тЪк | Not modeled (`graphql-default` / `hasura-default` have no effect). |

> **Migration note:** Hasura's `pool_settings` / `isolation_level` /
> `use_prepared_statements` blocks are valid metadata and parse without error, but
> Constellation does not act on them тАФ move pool tuning into the connection URL
> (e.g. `?pool_max_conns=50&pool_max_conn_lifetime=1h`).

## Table tracking & configuration

```yaml
table:
  name: users
  schema: public
is_enum: false
configuration:
  custom_name: users
  custom_root_fields:
    select: users
    select_by_pk: usersByPk
    # ...
  column_config:
    user_id:
      custom_name: userId
```

| Field | Status | Notes |
|---|---|---|
| `table.name`, `table.schema` | тЬЕ | `schema` honored on Postgres (multi-schema); SQLite is single-schema. |
| `is_enum` | тЬЕ | Marks an enum-mapping table. Requires one PK column (value) + optional second column (description); no other columns. |
| `configuration.custom_name` | тЬЕ | Overrides the generated GraphQL type name. |
| `configuration.custom_root_fields.*` | тЬЕ | All eleven overrides applied: `select`, `select_by_pk`, `select_aggregate`, `select_stream`, `insert`, `insert_one`, `update`, `update_by_pk`, `update_many`, `delete`, `delete_by_pk`. |
| `configuration.column_config.<col>.custom_name` | тЬЕ | Renames a column in the GraphQL schema (and threads through to relationships/queries). |
| `configuration.column_config.<col>.comment` | тЪк | Dropped. |
| `configuration.comment` | тЪк | Table comment is dropped. |
| `configuration.identifier` | тЪк | Dropped. |
| `apollo_federation_config` | тЭМ | No Apollo Federation support. |

## Permissions

Declared per table under `select_permissions`, `insert_permissions`,
`update_permissions`, `delete_permissions`, each a list of `{ role, permission }`.
The `admin` role bypasses permissions entirely. Session variables
(`X-Hasura-User-Id`, тАж) in `filter`/`check`/`set` are substituted as
parameterized SQL values.

### Select permission

| Field | Status | Notes |
|---|---|---|
| `columns` | тЬЕ | Gates column visibility. Accepts Hasura's `columns: '*'` all-columns shorthand and explicit column lists. |
| `filter` | тЬЕ | Row filter; AND-ed into every `WHERE`. Full Hasura boolean-expression syntax. |
| `allow_aggregations` | тЬЕ | Gates the `_aggregate` root field for the role. |
| `limit` | тЪк | **Not enforced.** A per-role row `limit` is parsed away and has no effect тАФ enforce row caps another way. |
| `query_root_fields` | тЪк | Cannot restrict which query root fields a role sees. |
| `subscription_root_fields` | тЪк | Same, for subscriptions. |
| `computed_fields` | тЪк | Computed fields are unsupported entirely. |

### Insert permission

| Field | Status | Notes |
|---|---|---|
| `columns` | тЬЕ | Gates which columns the role may set. Accepts Hasura's `columns: '*'` all-columns shorthand and explicit column lists. |
| `check` | тЬЕ | Per-row boolean check; rows failing it are rejected (all-or-nothing). Evaluated against the input payload by default and switched to a post-INSERT check (against `RETURNING *`) when the predicate references a column whose final value is only known after the row exists тАФ generated columns, identity columns (Postgres `GENERATED AS IDENTITY`, SQLite `INTEGER PRIMARY KEY` rowid alias), or a DB-defaulted column the payload omits. Enforced on Postgres via `RETURNING` + `constellation_throw_error`. |
| `set` | тЬЕ | Column presets (incl. session variables) forcibly written on every insert. |
| `backend_only` | тЪк | Not honored тАФ the mutation is exposed regardless. |
| `validate_input` | тЪк | No input-validation webhook. |

### Update permission

| Field | Status | Notes |
|---|---|---|
| `columns` | тЬЕ | Gates which columns the role may modify. Accepts Hasura's `columns: '*'` all-columns shorthand and explicit column lists. If a role has **no** updatable columns, the update mutations are omitted (see [`KNOWN_DIFFERENCES.md`](../../KNOWN_DIFFERENCES.md)). |
| `filter` | тЬЕ | Pre-condition: which rows are visible to update. |
| `check` | тЬЕ | Post-condition on the updated row; violations rejected. |
| `set` | тЬЕ | Column presets forcibly written on every update. |
| `backend_only` | тЪк | Not honored. |
| `validate_input` | тЪк | Not honored. |

### Delete permission

| Field | Status | Notes |
|---|---|---|
| `filter` | тЬЕ | Which rows the role may delete. |
| `backend_only` | тЪк | Not honored. |
| `validate_input` | тЪк | Not honored. |

### Other permission features

| Hasura feature | Status | Notes |
|---|---|---|
| Permission `comment` | тЪк | Dropped. |
| Inherited roles | тЭМ | See [top-level](#top-level-metadata). |

## Relationships

Declared on the parent table.

### Object & array relationships

```yaml
object_relationships:
  - name: user
    using:
      foreign_key_constraint_on: user_id          # column on this table

array_relationships:
  - name: sessions
    using:
      foreign_key_constraint_on:
        column: user_id
        table: { name: sessions, schema: auth }    # FK on the other table
```

| Form | Status | Notes |
|---|---|---|
| `foreign_key_constraint_on: <column>` (string) | тЬЕ | FK on this table. |
| `foreign_key_constraint_on: { column, table }` | тЬЕ | FK on the remote table pointing back. |
| `foreign_key_constraint_on: [col1, col2]` / `{ columns: [тАж], table }` | тЬЕ | **Composite (multi-column) foreign keys are supported** in both the array form (`[col1, col2]`, forward FK on this table) and the object form (`{ columns, table }`, reverse FK on the target table). They are parsed, resolved against the introspected FKs, and build working multi-column relationships (forward and reverse). The only narrow caveat is that the bare array form carries no explicit target-table hint, so its target is inferred from the matching introspected FK rather than declared. |
| `using.manual_configuration` (`remote_table`, `column_mapping`) | тЬЕ | Multi-entry `column_mapping` forms a composite join key. |
| `using.manual_configuration.insertion_order` (array rels) | тЪк | Dropped. |
| relationship `comment` | тЪк | Dropped. |

### Remote relationships (table тЖТ other source / remote schema)

```yaml
remote_relationships:
  - name: orders
    definition:
      to_source:
        source: warehouse
        table: { name: orders, schema: public }
        relationship_type: array        # or "object"
        field_mapping: { id: customer_id }
  - name: weather
    definition:
      to_remote_schema:
        remote_schema: weather_api
        lhs_fields: [city]
        remote_field:
          forecast:
            arguments: { city: $city }
```

| Field | Status | Notes |
|---|---|---|
| `definition.to_source` (`source`, `table`, `relationship_type`, `field_mapping`) | тЬЕ | Cross-database relationship, resolved post-execution by `controller/resolver`. |
| `definition.to_remote_schema` (`remote_schema`, `lhs_fields`, `remote_field`) | тЬЕ | Database тЖТ remote-schema relationship. Column renames thread through `lhs_fields`/`remote_field` arguments. |

See [`docs/developers/remote-relationships.md`](../developers/remote-relationships.md)
for the planner/resolver mechanics.

## Functions

Postgres only (SQLite has no function support тАФ gated by `SupportsFunctions`).

```yaml
function:
  name: search_articles
  schema: public
configuration:
  custom_name: searchArticles
  custom_root_fields:
    function: searchArticles
    function_aggregate: searchArticlesAggregate
  session_argument: hasura_session
  exposed_as: query            # or "mutation"
permissions:
  - role: user
```

| Field | Status | Notes |
|---|---|---|
| `function.name`, `function.schema` | тЬЕ | Must return a table type or `SETOF` composite. |
| `configuration.custom_name` | тЬЕ | Base name for generated fields. |
| `configuration.custom_root_fields.function` / `.function_aggregate` | тЬЕ | Field-name overrides. |
| `configuration.exposed_as` | тЬЕ | `query` or `mutation`. Defaults from volatility (`IMMUTABLE`/`STABLE` тЖТ query, `VOLATILE` тЖТ mutation). |
| `configuration.session_argument` | тЬЕ | Named arg receives session vars as JSON; hidden from the schema, injected at execution. |
| `permissions` (`role`) | тЬЕ | Roles allowed to call the function. **Permissions are never inferred** (`HASURA_GRAPHQL_INFER_FUNCTION_PERMISSIONS=false`, not configurable) тАФ they must be set explicitly. See [`KNOWN_DIFFERENCES.md`](../../KNOWN_DIFFERENCES.md). |
| `configuration.response_object` / per-arg customization | тЪк | Not modeled. |
| function `comment` | тЪк | Dropped. |

Single-row (`RETURNS <table>`, not `SETOF`) functions intentionally omit
`where`/`order_by`/`limit` and the `_aggregate` field тАФ see `KNOWN_DIFFERENCES.md`.

## Remote schemas

```yaml
- name: weather_api
  definition:
    url_from_env: WEATHER_API_URL    # or `url:` literal
    timeout_seconds: 60
    forward_client_headers: true
    headers:
      - name: X-Api-Key
        value_from_env: WEATHER_API_KEY
  permissions:
    - role: user
      definition:
        schema: |
          # role-scoped SDL with @preset(...) directives
```

| Field | Status | Notes |
|---|---|---|
| `definition.url` / `definition.url_from_env` | тЬЕ | Endpoint, with `{{VAR}}`/`from_env` interpolation. |
| `definition.timeout_seconds` | тЬЕ | HTTP client timeout (default 60s). |
| `definition.headers` (`value` / `value_from_env`) | тЬЕ | Static headers attached to every upstream request. |
| `definition.forward_client_headers` | тЬЕ | Forwards the incoming client headers upstream. |
| `permissions` (`role` тЖТ `definition.schema` SDL) | тЬЕ | Per-role SDL. Admin always introspects the live schema; other roles get the SDL. |
| `@preset(value: тАж)` directive in role SDL | тЬЕ | Hides an argument and injects a literal or session variable. See [`remote-schema.md`](./remote-schema.md). |
| `remote_relationships` (`type_name` тЖТ `to_source`) | тЬЕ | Remote-schema-type тЖТ database-source relationships. |
| `remote_relationships` тЖТ `to_remote_schema` | тЭМ | Remote-schema тЖТ remote-schema relationships are not modeled (only `to_source`). |
| `definition.customization.root_fields_namespace` | тЬЕ | Wraps every root field under a single `<namespace>Query` / `<namespace>Mutation` / `<namespace>Subscription` field, matching Hasura's remote-schema naming. |
| `definition.customization.type_names` (`prefix`, `suffix`, `mapping`) | тЬЕ | Renames types; `mapping` overrides prefix/suffix for the specific names it lists. |
| `definition.customization.field_names` (per-type field renames) | тЭМ | **Rejected at startup** with an error тАФ unlike most unsupported fields this is *not* silently dropped, because the customized schema would advertise renamed fields the execution path cannot reverse. Remove `field_names` to start. |
| `definition.introspection_cache_ttl` | тЪк | Dropped. |
| `comment` | тЪк | Parsed but not surfaced. |

Full details: [`docs/user/remote-schema.md`](./remote-schema.md) and
[`docs/developers/remote-schemas.md`](../developers/remote-schemas.md).

## Entirely unsupported feature areas

These Hasura metadata sections have **no representation** in Constellation. When
present in a Hasura metadata file or `hdb_metadata` blob they are dropped on load
(no error); when configured through Hasura's Metadata API they have no effect on
what Constellation serves. **Exception:** event triggers are a partial case тАФ
on the DB metadata source their config is stored and round-trips through
`export_metadata` (no runtime fires them); see the row below.

| Hasura feature | Metadata operations | Status |
|---|---|---|
| **Actions** | `create_action`, `create_action_permission`, тАж | тЭМ |
| **Custom types** | `set_custom_types` | тЭМ |
| **Event triggers** | `pg_create_event_trigger`, `pg_delete_event_trigger`, тАж | ЁЯЯб тАФ **Metadata authoring only (DB source).** `pg_create_event_trigger`/`pg_delete_event_trigger` are stored and the `event_triggers` key round-trips through `export_metadata`. No event-delivery runtime executes them (triggers never fire), and the file/YAML source still drops them (the struct field is tagged `yaml:"-"`). |
| **Cron / scheduled triggers** | `create_cron_trigger`, `create_scheduled_event` | тЭМ |
| **Query collections** | `create_query_collection`, `add_query_to_collection` | тЭМ |
| **Allowlist** | `add_collection_to_allowlist`, тАж | тЭМ |
| **RESTified endpoints** | `create_rest_endpoint` | тЭМ |
| **Inherited roles** | `add_inherited_role` | тЭМ |
| **Computed fields** | `*_add_computed_field` | тЭМ (on the roadmap) |
| **API limits** | `set_api_limits` | тЭМ |
| **Network / TLS allowlist** | `add_host_to_tls_allowlist` | тЭМ |
| **Metrics config** | `set_metrics_config` | тЭМ |
| **OpenTelemetry** | `set_opentelemetry_config` | тЭМ |
| **GraphQL introspection options** | `set_graphql_introspection_options` | тЭМ |
| **Logical models** | `*_track_logical_model` | тЭМ |
| **Native queries** | `*_track_native_query` | тЭМ |
| **Stored procedures** (MSSQL) | `mssql_track_stored_procedure` | тЭМ (no MSSQL backend) |
| **Metadata Management HTTP API** | `POST /v1/metadata` (`export_metadata`, `replace_metadata`, `clear_metadata`, `reload_metadata`, тАж) | тЪая╕П тАФ **When an in-process metadata Store is active (database metadata source)** it is the source of truth: `export_metadata` / `replace_metadata` / `clear_metadata` / `reload_metadata` are served **natively** against `hdb_catalog.hdb_metadata` (with optimistic concurrency on `resource_version`), regardless of upstream config; the proxy is only a per-op fallback for ops with no native handler. **When there is no Store:** with `--hasura-upstream-url` set, every op (including `export_metadata`) is proxied to that upstream so the CLI/dashboard exportтЖТeditтЖТreplace cycle is consistent; with no upstream configured, `export_metadata` is served natively from the current snapshot and other ops return `not-supported`. **File-source caveat:** when metadata is loaded from a local YAML file (dev mode, no Store), `export_metadata` returns a best-effort inspection view of the recognised fields, not a lossless re-encoding of the source file тАФ unmodeled top-level keys (e.g. `actions`, `cron_triggers`) and some scalar defaults are dropped. ## Sharp edges, in one place

- **Nothing is rejected.** Unsupported sections and ignored fields load silently.
  If a permission `limit`, a `pool_settings` block, or an `actions` list seems to
  have "no effect," that is expected тАФ Constellation never read it.
- **`limit` on select permissions does nothing.** Enforce row caps another way.
- **Pool tuning goes in the connection URL,** not `pool_settings`.
- **Composite foreign keys are supported** via `foreign_key_constraint_on` in
  both its array form (`[col1, col2]`) and object form (`{ columns, table }`),
  building working multi-column relationships; `manual_configuration` with a
  multi-entry `column_mapping` also works.
- **Schema `customization` is applied** тАФ both source-level
  (`sources[].customization`) and remote-schema
  (`remote_schemas[].definition.customization`): root-field
  namespacing/prefix/suffix and `type_names` renaming. Two carve-outs: database
  `naming_convention` is ignored, and remote-schema `field_names` is **rejected at
  startup** (not silently dropped). Combining `customization` with remote
  relationships on the same source is not yet handled.
- **`kind` must be `postgres` or `sqlite`;** other backends fail at startup.

## Metadata-authoring parity tests

`integration/metadata_parity_test.go` is a differential test that applies the
same `/v1/metadata` op to **both** Hasura and Constellation and asserts the
results are equivalent, in four layers:

- **Layer A тАФ response parity:** matching HTTP status class and, on error, the
  same Hasura error `code`. Skipped for ops where Constellation intentionally
  diverges (idempotent re-apply returns `200` + an idempotency code where Hasura
  returns `4xx`; see `KNOWN_DIFFERENCES.md`).
- **Layer B тАФ exported metadata:** `export_metadata` from both engines, flattened
  to content-addressed leaf sets and compared as a strict set difference. This is
  the primary assertion.
- **Layer C тАФ schema delta:** for ops that change the GraphQL surface, the
  per-engine SDL delta-vs-baseline (via the CLI `schema dump`/`schema diff`), so
  pre-existing baseline divergences (`integration/schema.*.diff`) don't cause
  false failures. **Opt-in and OFF by default** тАФ it dumps the full SDL via the
  CLI several times per case and is far slower than Layers A/B, so it only runs
  when `PARITY_SCHEMA_CHECK=1` is set.
- **Layer D тАФ query execution:** for cases that define a `query`, the harness runs
  it against both engines' GraphQL endpoints after the op and diffs the
  responses (honouring optional `queryRole`/`queryWantErr`), proving the op
  produced an equivalent *runtime* schema and not just equivalent metadata.
  Per-case and opt-in тАФ it runs only for cases that set a `query`.

**Isolation.** The two engines cannot share one `hdb_catalog.hdb_metadata` row,
so the harness gives Constellation its own metadata DB (`cstl`, created by
`make parity-env-up`) while it still introspects the same `local` data DB as
Hasura тАФ so their schemas match. Before every case both engines are reset to
Hasura's live baseline via `replace_metadata`, making cases order-independent.

**Strictness тАФ divergence is an error.** Constellation aims to be a drop-in
Hasura replacement, so Layer B deliberately does **not** normalize away
representational differences (no dropping of `comment:""`, empty objects, etc.).
The only neutral step is content-addressing array elements, since Hasura does not
guarantee element order for metadata collections. Two things follow:

- A dedicated `roundtrip_baseline_fidelity` subtest resets both engines to the
  same baseline and fails on *any* divergence between their exports тАФ these are
  Constellation round-trip-fidelity gaps (e.g. it re-serializes
  `foreign_key_constraint_on: {column: "x"}` as `{columns: ["x"]}`, or a
  permission `columns: "*"` as `["*"]`). They are properties of the snapshot, not
  of any op.
- Each op case subtracts that baseline divergence and fails if the op introduces
  **any new** divergence тАФ that is op-effect parity.

Genuinely acceptable divergences (if any) are added to the explicit,
comment-documented `justifiedDivergence` allowlist in `metadata_parity_test.go`,
which is **empty by default**. Nothing is swept under the rug: an unexpected
difference fails the suite until it is either fixed in Constellation or
explicitly justified.

**Running it:**

```bash
make dev-env-integration-up      # Hasura + local data DB + auth/storage/functions
make build-docker-image          # constellation:0.0.0-dev
make parity-env-up               # creates cstl, starts constellation-parity on :8001
cd integration && go test -run TestMetadataParity -v ./...
make parity-env-down
```

This runs Layers A and B only. To **also** run the schema-delta layer (Layer C),
set `PARITY_SCHEMA_CHECK=1` тАФ it is opt-in and OFF by default because it dumps the
full SDL via the CLI several times per case and is slow:

```bash
cd integration && PARITY_SCHEMA_CHECK=1 go test -run TestMetadataParity -v ./...
```

When the parity Constellation (`:8001`) is not running, the test **skips** with a
pointer to `make parity-env-up` rather than failing.

Not yet covered (each needs a fixture or has a documented semantic gap, called
out in the test file): `pg_set_table_is_enum`, remote relationships, and the read
ops `pg_suggest_relationships` / `pg_get_viewdef`.
